# 成员赛事奖项申报工作流

版本：`v0.1`
日期：`2026-06-09`
状态：`已进入实现`

## 1. 背景

当前系统已经具备：

- `Contest / ContestTeam / ContestResult` 正式赛事档案
- `ACM Rating / ACM Level` 计算逻辑
- 管理员维护赛事、队伍和成绩的能力

但仍缺少一条“成员主动导入自己的赛事获奖信息”的入口。实际使用中，很多赛事信息最先掌握在成员手里，因此需要补上一条类似“活动申请”的流程：

- 由 `Member` 先提交赛事奖项申报
- 由 `Admin` 审核并可修改
- 通过后再写入正式赛事档案

## 2. 目标

本流程需要满足：

1. 成员可以提交自己的赛事与获奖信息
2. 管理员拥有最终编辑权和决定权
3. 未审核通过前，申报信息不能直接影响 `Rating / Level`
4. 审核通过后，系统自动同步正式 `Contest / Team / Result`
5. 审核记录、驳回说明和落库结果可追踪

## 3. 设计边界

本轮采用“双层数据”设计：

- `ContestSubmission`
  成员提交的申报草稿与审核记录
- `Contest / ContestTeam / ContestResult`
  管理员确认后的正式竞赛数据

这意味着：

- 成员不会直接写正式赛事表
- 管理员审核页中可以修改成员原始申报
- 只有审核通过后，才会落正式赛事数据并参与评分

## 4. 角色流程

### 4.1 Member

成员可以：

- 新建赛事奖项申报
- 查看自己的申报记录
- 查看审核状态与审核说明

成员提交时填写的信息包括：

- 赛事名称
- 赛事系列
- 赛季/学年
- 阶段
- 比赛日期
- 主办方
- 赛事级别
- 队伍名称
- 系统内队友
- 未建档队友
- 奖项类型
- 奖项展示文案
- 名次描述
- 证据链接
- 补充说明

### 4.2 Admin

管理员可以：

- 查看全部成员申报
- 按状态/关键词筛选申报
- 在审核页中修改成员提交内容
- 选择“新建正式赛事”或“合并到已有赛事”
- 审核通过或驳回

### 4.3 System

系统在审核通过后自动：

1. 创建或复用正式 `Contest`
2. 创建或复用正式 `ContestTeam`
3. 创建或复用正式 `ContestResult`
4. 将成绩标记为 `verified`
5. 重算相关成员 `Rating / Level`

## 5. 数据说明

### 5.1 ContestSubmission

职责：

- 保存成员提交的原始赛事申报
- 保存审核状态和审核说明
- 记录与正式赛事数据的映射关系

关键字段：

- `applicant`
- `contest_name`
- `contest_series`
- `contest_season`
- `contest_stage`
- `contest_date`
- `contest_level`
- `team_name`
- `team_members`
- `external_teammates`
- `award_type`
- `award_label`
- `rank_label`
- `evidence_url`
- `submission_note`
- `review_status`
- `review_note`
- `reviewed_by`
- `reviewed_at`
- `resolved_contest`
- `resolved_team`
- `resolved_result`

### 5.2 正式数据同步策略

审核通过时：

- 如果管理员指定了 `linked_contest`
  则并入已有赛事
- 否则
  使用审核后的申报内容创建或更新正式赛事

队伍层：

- 复用同赛事下同名队伍，或创建新队伍
- `team_members` 写入正式队伍成员
- `external_teammates` 写入正式队伍的未建档成员字段

成绩层：

- 复用同赛事同队伍的成绩记录，或创建新记录
- 审核通过后自动 `verified`

## 7. 评分影响

- `ContestSubmission` 本身不参与评分
- 只有审核通过并同步为正式 `ContestResult` 后，才参与 `Rating / Level`
- 驳回状态不会影响成员竞赛档案

## 8. 当前实现假设

本轮默认以下假设：

1. 成员提交的是“自己的获奖申报”，不提供成员端直接编辑正式赛事数据的能力
2. 如果同一场赛事已存在，管理员可以在审核页中手动并入已有赛事
3. 未建档队友先以文本形式保存，不强制要求全部先注册为系统成员
4. 审核通过后，正式结果以管理员审核页中的内容为准，而不是成员最初填写内容

## 9. 后续可扩展

后续可以继续增强：

- 成员补充上传附件
- 同一赛事下多次申报的合并提示
- 审核通过后的通知提醒
- 成员撤回待审核申报
- 批量审核与批量导入
