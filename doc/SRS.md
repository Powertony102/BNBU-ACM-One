# One BNBU-ACM 活动签到系统 SRS

版本：`v0.3`
日期：`2026-06-09`
状态：需求确认稿，适用于 Prototype Modeling

## 1. 项目概述

### 1.1 项目名称

`One BNBU-ACM`

### 1.2 项目目标

One BNBU-ACM 是面向大学生程序设计竞赛团队的活动签到系统，目标是：

1. 为队员提供清晰、便捷的活动参与与签到入口。
2. 为管理员提供活动创建、队员管理、签到管理与统计能力。
3. 通过 `ACM Star` 展示队员近期活跃状态，强化团队参与感。
4. 基于 Prototype Modeling，先快速形成可演示原型，再逐步迭代。

### 1.3 原型范围

第一版原型聚焦于以下能力：

- 账号登录与权限控制
- 队员个人中心
- 活动发布与管理
- 活动签到
- 二维码签到入口
- `ACM Star` 点亮逻辑
- 基础统计面板

以下内容暂不纳入第一版原型：

- 报名系统
- 比赛成绩管理
- 复杂积分和排行体系
- 与校园统一身份认证深度集成
- 多组织、多校区支持

## 2. 术语定义

- **队员 Member**：ACM 团队成员，可以登录系统、查看活动、签到、维护个人资料。
- **普通管理员 Admin**：负责管理活动、签到和队员资料，但不负责系统级权限配置。
- **超级管理员 Super Admin**：拥有系统级管理能力，可管理管理员账号、角色权限和系统参数。
- **系统 System**：自动执行登录鉴权、状态计算、二维码校验、签到判定、日志记录等内部逻辑。
- **活动 Event**：一次 ACM 相关线下或线上活动，如训练、讲座、分享会、比赛说明会等。
- **签到 Check-in**：队员对某次活动的出席登记记录。
- **签到二维码 QR Check-in Code**：由系统为活动生成的签到入口二维码，队员扫码后进入该活动的签到流程。
- **ACM Star**：队员近期是否参加过 ACM 活动的状态标识。
- **近期窗口 Recent Window**：用于判定 `ACM Star` 是否点亮的时间范围，默认建议为 `30` 天。
- **活动申请 Event Application**：由 `Member` 发起、等待 `Admin` / `Super Admin` 审核的一次活动举办申请。
- **事件级签到管理 Event-scoped Check-in Management**：仅针对单次已获批活动开放的签到二维码生成、签到关闭和签到记录查看能力。

## 3. 角色与权限

### 3.1 Actor 列表

系统当前确认的 actor 如下：

1. `Member`
2. `Admin`
3. `Super Admin`
4. `System`

### 3.2 角色权限边界

#### Member

- 可以登录和退出系统
- 可以查看自己的活动与签到数据
- 可以修改自己的个人资料
- 可以通过二维码或页面入口完成签到
- 可以提交自己想举办的活动申请
- 在自己的活动申请通过后，仅管理该活动的签到
- 不能管理其他成员和活动

#### Admin

- 拥有活动管理权限
- 可以审核成员活动申请
- 拥有签到记录管理权限
- 可以查看和编辑队员资料
- 可以停用普通队员账号
- 可以查看统计结果和日志
- 不能管理管理员分级与系统级配置

#### Super Admin

- 拥有 `Admin` 的全部权限
- 可以创建、启用、停用管理员账号
- 可以设定管理员权限级别
- 可以配置系统参数，例如 `ACM Star` 近期窗口
- 可以查看完整操作日志

#### System

- 负责鉴权、权限校验、签到规则校验、二维码校验、星标计算、状态刷新和日志记录

## 4. Use Case 需求

本系统以 actor 的 use case 为需求主线。

### 4.1 Member Use Cases

#### UC-M01 登录系统

- 目标：队员使用账号进入系统
- 前置条件：账号已存在且处于启用状态
- 基本流程：
  1. 输入用户名和密码
  2. 系统校验账号与密码
  3. 系统识别角色并进入队员首页

#### UC-M02 查看个人首页

- 目标：查看个人当前活跃状态和最近活动情况
- 页面应展示：
  - `ACM Star` 状态
  - 最近参加的活动
  - 待签到或近期活动
  - 个人累计签到次数

#### UC-M03 查看个人签到历史

- 目标：查看自己参加过的活动和签到记录
- 队员可以按时间查看历史签到

#### UC-M04 查看活动详情

- 目标：查看某活动的时间、地点、签到开放状态和描述

#### UC-M05 修改个人资料

- 目标：维护自己的基础信息
- 可修改字段建议包括：
  - 邮箱
  - 手机号
  - 专业
  - 班级
- 不可直接修改字段建议包括：
  - 学号
  - 角色
  - 账号状态

#### UC-M06 通过页面签到

- 目标：对开放中的活动进行签到
- 前置条件：
  - 队员已登录
  - 活动处于可签到状态
- 系统应记录签到时间、签到方式和签到结果

#### UC-M07 通过二维码签到

- 目标：通过扫描活动二维码进入签到流程
- 基本流程：
  1. 队员扫描活动二维码
  2. 系统打开签到链接
  3. 若未登录，系统引导先登录
  4. 登录成功后返回该活动签到页面
  5. 系统校验签到时间窗口、活动状态和是否重复签到
  6. 签到成功后给出反馈

#### UC-M08 提交活动申请

- 目标：成员为 lecture、专题分享等活动提交举办申请
- 基本流程：
  1. 成员填写活动名称、类型、说明、地点和时间信息
  2. 系统保存申请并标记为 `待审核`
  3. 成员可查看审核状态与审核意见

#### UC-M09 管理自己获批活动的签到

- 目标：在活动申请审核通过后，仅针对该次活动执行签到管理
- 前置条件：
  - 该活动由当前成员提交
  - 当前活动已被 `Admin` 或 `Super Admin` 审核通过
- 可执行操作：
  - 查看该活动签到详情
  - 生成该活动二维码签到入口
  - 关闭该活动签到
  - 查看更完整签到名单
  - 为该活动手动补签
  - 撤销该活动签到记录
- 权限边界：
  - 不获得全局后台权限
  - 不可编辑其他活动
  - 不可审核其他成员申请

### 4.2 Admin Use Cases

#### UC-A01 登录后台

- 目标：管理员进入管理端

#### UC-A02 查看管理首页

- 目标：查看系统运营概况
- 页面建议展示：
  - 近期活动数
  - 活跃队员数
  - 今日签到情况
  - Star 点亮人数

#### UC-A03 创建活动

- 目标：新增一场活动
- 关键字段：
  - 活动名称
  - 活动类型
  - 活动描述
  - 活动地点
  - 开始时间
  - 结束时间
  - 签到开始时间
  - 签到结束时间
  - 可选指定一名 `Member` 作为该活动的签到管理员

#### UC-A04 编辑活动

- 目标：修改活动信息

#### UC-A05 发布或关闭活动

- 目标：控制活动是否可被队员看到和签到

- 删除后：
  - 活动不再对普通队员开放
  - 已生成签到入口会失效
  - 历史签到记录仍保留用于审计和统计

#### UC-A06 生成活动签到二维码

- 目标：为某活动生成专属二维码
- 系统应生成唯一签到链接
- 二维码应关联具体活动
- 后续可以扩展过期时间或一次性签入策略

#### UC-A07 查看活动签到名单

- 目标：查看某活动的签到情况
- 可查看：
  - 已签到名单
  - 未签到名单
  - 签到时间
  - 签到方式

#### UC-A08 手动补签

- 目标：为特殊情况未扫码签到的队员补录签到

#### UC-A09 撤销签到

- 目标：纠正错误签到记录

#### UC-A10 新增队员

- 目标：创建队员账号和资料

#### UC-A11 编辑队员资料

- 目标：维护队员基础信息

#### UC-A12 停用或启用队员账号

- 目标：控制队员是否能登录和签到

#### UC-A13 重置队员密码

- 目标：在队员遗忘密码时协助恢复访问

#### UC-A14 查看队员参与历史

- 目标：查看某队员参加活动和签到情况

#### UC-A15 查看统计信息

- 目标：从活动和个人两个维度查看数据

#### UC-A16 审核成员活动申请

- 目标：审核成员提交的活动举办申请
- 基本流程：
  1. 管理员查看待审核活动
  2. 填写审核说明并选择“通过”或“驳回”
  3. 系统记录审核人、审核时间、审核说明与审核结果
  4. 若审核通过，系统自动开放该活动，并授予申请人该活动的事件级签到管理权限

## 5. 功能性需求

### 5.1 认证与权限

- 系统应支持用户名密码登录
- 系统应支持角色区分：`Member`、`Admin`、`Super Admin`
- 系统应支持退出登录
- 系统应拒绝停用账号登录

### 5.2 队员资料

- 队员应可查看自己的个人资料
- 队员应可修改允许自助修改的资料字段
- 管理员应可维护队员资料
- 系统应保留学号等关键字段的唯一性约束

### 5.3 活动管理

- 管理员应可创建、编辑、发布、关闭、作废活动
- 队员应可提交活动申请
- 管理员应可审核成员提交的活动申请
- 管理员创建或编辑活动时应可指定一名 `Member` 拥有该活动的签到管理权限
- 管理员在指定签到管理员时应支持通过姓名、邮箱、学号或用户名搜索筛选
- 活动应具备唯一标识
- 活动应支持签到时间窗口设置
- 活动应支持签到二维码生成
- 活动应支持编辑与删除入口

### 5.4 签到管理

- 队员应可自主签到
- 队员应可通过二维码进入签到流程
- 管理员应可补签和撤销签到
- 获批活动的申请人应可仅针对该活动管理签到二维码与签到关闭
- 获批活动的申请人应可查看完整签到名单，并可仅针对该活动执行补签和撤销
- 系统应记录签到方式，例如：
  - `manual`
  - `web`
  - `qr`

### 5.5 ACM Star

- 系统应根据“近期窗口内是否存在有效签到”决定星标状态
- 默认规则建议为：最近 `30` 天内至少签到一次则点亮
- `Super Admin` 应可调整近期窗口天数

### 5.6 查询与统计

- 系统应支持按活动查看签到名单
- 系统应支持按队员查看参与历史
- 系统应支持基础统计：
  - 活动签到人数
  - 队员累计参与次数
  - 近期活跃队员数
  - Star 点亮人数

## 6. 非功能性需求

### 6.1 技术约束

- 后端使用 `Python + Django`
- 数据库优先使用 `SQLite`
- 前端原型建议采用 `Django Template + 少量 JavaScript`
- 后续如有需要可逐步演进到 `Vue.js`

### 6.2 可用性

- 原型应适合桌面端浏览器使用
- 扫码签到页面应兼容移动端浏览器
- 操作反馈应清晰，例如签到成功、重复签到、二维码无效、签到未开放

### 6.3 安全性

- 密码必须加密存储
- 角色权限必须在后端校验
- 二维码不得直接暴露管理能力
- 审计日志应记录敏感操作

### 6.4 可维护性

- 领域模型应保持清晰
- 权限设计应便于从 `Admin` 扩展为多级管理体系
- 二维码签到应尽量设计为独立模块，方便后续替换实现

## 8. 核心业务规则

1. 一个队员对同一活动原则上只能存在一条有效签到记录。
2. 队员停用后不得登录、签到或修改个人资料。
3. 管理员账号分级必须由 `Super Admin` 管理。
4. 队员只能修改被允许开放的个人资料字段。
5. 扫码签到时，如果用户未登录，系统必须在登录后返回原签到页面。
6. 二维码对应的签到入口必须绑定具体活动。
7. 活动未发布、已结束、已关闭签到、二维码失效等情况下，不应允许普通签到成功。
8. `ACM Star` 的判定依据是有效签到，而不是报名。

## 9. 原型实现建议

建议按以下顺序推进：

1. 确定 `User`、`MemberProfile`、`AdminProfile`、`Event`、`CheckInRecord` 数据模型。
2. 实现登录、角色鉴权和基础页面框架。
3. 实现管理员活动管理与队员管理。
4. 实现队员首页、个人资料和 `ACM Star` 展示。
5. 实现二维码生成与扫码签到流程。
6. 最后补充统计面板与审计日志。

## 10. 当前已确认事项

1. 队员可以修改自己的部分个人资料。
2. 管理员需要分级，至少区分 `Admin` 与 `Super Admin`。
3. 活动不设置报名流程，仅保留签到。
4. 系统应提供活动二维码，队员扫码后进入登录并签到流程。
