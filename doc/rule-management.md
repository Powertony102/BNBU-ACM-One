# 规则管理说明

日期：`2026-06-11`

## 1. 背景

管理端原先只有一个偏“系统参数”的 `Star` 规则页面，而 `Rating` 相关的赛事级别权重、奖项加分、定级阈值主要写死在代码里，导致：

- 超级管理员无法直接在后台调整 `Rating` 口径。
- 导航上看不到统一的“规则管理”入口。
- 规则修改后的影响范围不够直观，尤其是 `Rating -> Level` 的颜色梯度与阈值关系。

本轮调整把这类能力统一收口到新的“规则管理”分类中。

## 2. 信息架构

新的后台导航结构：

- `规则管理`
- `规则总览`
- `Star 规则`
- `Rating 规则`

其中：

- `Star 规则` 负责近期活跃窗口与签到二维码时效。
- `Rating 规则` 负责赛事权重、奖项加分、基础参赛分、定级阈值。
- `规则总览` 负责先看当前生效口径，再决定进入哪一页调整。

## 3. 主要 Use Cases

### 3.1 调整 Star 活跃窗口

适用于：

- 学期初希望把 `ACM Star` 点亮窗口从 `30` 天收紧到 `21` 天。
- 活动密度变化后，希望适当放宽或收紧近期活跃定义。

操作路径：

- `规则管理 -> Star 规则`

### 3.2 调整赛事权重

适用于：

- 希望国家级赛事相较校赛带来更高的 `Rating` 影响。
- 团队发现校赛/队内赛被高估或低估，需要统一修正默认权重。

操作路径：

- `规则管理 -> Rating 规则 -> 赛事级别权重`

说明：

- 若某场赛事勾选“跟随规则管理中的默认权重”，则会随着这里的配置变化自动更新。
- 若某场赛事取消该勾选，则保留自己的手动权重，不受默认权重影响。

### 3.3 调整奖项加分

适用于：

- 想提升金奖、银奖的区分度。
- 想降低“仅参赛”对总分的影响。

操作路径：

- `规则管理 -> Rating 规则 -> 奖项加分`

### 3.4 调整定级阈值

适用于：

- 一段时间后成员整体 `Rating` 上涨，需要重新划分 `Rookie / Solver / Expert` 边界。
- 希望通过颜色化预览快速判断各等级区间是否清晰。

操作路径：

- `规则管理 -> Rating 规则 -> Rating 定级阈值`

说明：

- 页面直接以等级颜色展示每一档。
- 每次输入阈值后，前端会即时刷新区间预览。

## 4. 实现说明

### 4.1 规则存储

本轮仍复用 `SystemSetting` 作为持久化入口，新增以下键：

- `rating_base_participation_score`
- `contest_level_weight_rules`
- `award_bonus_rules`
- `competition_level_threshold_rules`

好处：

- 无需引入额外后台 CRUD 模型。
- 默认值仍可保留在代码层，数据库仅存覆盖项。

### 4.2 赛事权重跟随模式

`Contest` 新增字段：

- `use_default_weight`

语义：

- `True`：赛事权重跟随全局级别规则。
- `False`：赛事权重为该赛事单独覆写值。

这让“调整全局规则”和“保留个别特殊赛事口径”可以同时成立。

### 4.3 Rating 重算策略

保存 `Rating` 规则后，系统会：

1. 更新全局规则配置。
2. 同步刷新所有 `use_default_weight=True` 的赛事权重。
3. 全量重算成员竞赛档案。

这样可以保证：

- 后台看到的新规则会立刻反映到成员 `Rating / Level`。
- 无需手动逐条编辑赛事或成绩。

## 5. UI 设计要点

本轮重点处理了以下可用性问题：

- 所有规则输入框默认预加载当前生效值。
- `Rating` 阈值以颜色卡片直接展示，不再只给出一组抽象数字。
- 赛事创建/编辑页支持“跟随默认权重 / 手动覆写”双模式。
- 成绩录入页直接展示当前默认加分口径，并给出即时预估说明。

## 6. 后续可扩展点

如果后面要继续增强，可以优先考虑：

- 让 `Rating` 规则页支持“恢复默认值”。
- 引入时间衰减规则的后台化配置。
- 在规则总览页增加“本次修改将影响多少成员/赛事”的预估信息。
