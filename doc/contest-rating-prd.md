# ACM 赛事等级体系 PRD 草案

版本：`v0.1`
日期：`2026-06-09`
状态：`产品草案`

## 1. 目标

### 1.1 背景

当前系统中的 `ACM Star` 已经被定义为“近期活跃状态”模块，其核心依据是最近一段时间内的有效签到记录，用于展示队员最近是否持续参与活动。

随着系统准备纳入 `CCPC`、`ICPC`、校赛、选拔赛等正式赛事，现有 `ACM Star` 已不足以表达队员的竞赛积累、获奖情况与长期成长轨迹。因此需要新增一套面向赛事成绩的长期评价体系。

### 1.2 产品目标

本方案新增一套与 `ACM Star` 并行但职责清晰区分的竞赛能力模块，包括：

- `ACM Rating`：用统一积分表示队员的赛事积累
- `ACM Level`：用等级称号与颜色名表示当前竞赛梯队
- `Contest / Result`：用赛事与获奖记录作为积分与等级的依据

该模块需要实现以下目标：

1. 区分“近期活跃”与“长期成绩”两种不同维度，避免继续堆叠新的徽章体系。
2. 为成员提供清晰、稳定、可传播的竞赛身份展示方式。
3. 为管理员提供可维护的赛事录入、成绩认证和天梯统计能力。
4. 控制第一版实现复杂度，避免依赖难以持续维护的数据字段。

### 1.3 定位边界

- `ACM Star` 负责回答：队员最近是否活跃。
- `ACM Rating / Level` 负责回答：队员当前竞赛积累与实力大致处于什么梯队。
- `赛事成绩记录` 负责回答：该等级和积分来自哪些比赛与奖项。

### 1.4 第一版不涵盖

为保证落地性，第一版明确不纳入以下内容：

- 不记录 `解题数量`
- 不依赖外部榜单自动抓取
- 不实现复杂的实时 Elo 对战模型
- 不引入第二套类似 `ACM Pass` 的独立徽章系统

## 2. 数据模型

### 2.1 总体思路

在现有 `MemberProfile` 基础上，新增“赛事本体”“参赛队伍”“成绩记录”“等级快照”四类数据。其职责分层如下：

- `Contest`：描述一场赛事是什么
- `ContestTeam`：描述谁以什么队伍参加了赛事
- `ContestResult`：描述该队伍最终获得了什么结果
- `MemberCompetitionProfile`：描述某位成员当前汇总后的 `Rating / Level`

### 2.2 Contest

职责：

- 保存赛事基础信息
- 为成绩录入与展示提供统一归属
- 提供赛事级别与权重依据

建议字段：

- `name`
  赛事名称，例如 `CCPC 吉林省赛`
- `series`
  赛事系列，例如 `ICPC`、`CCPC`、`校赛`、`选拔赛`、`邀请赛`
- `season`
  学年或赛季，例如 `2026`
- `stage`
  阶段，例如 `校内选拔`、`省赛`、`区域赛`、`总决赛`
- `contest_date`
  比赛日期
- `organizer`
  主办方，可选
- `level`
  赛事级别，用于评分权重
- `weight`
  权重数值，可由系统根据级别默认生成，也可人工调整
- `status`
  状态，例如 `draft / published / archived`
- `description`
  补充说明，可选

建议枚举：

- `series`
  `icpc / ccpc / provincial / invitational / campus / selection / other`
- `level`
  `national / regional / provincial / campus / internal`

### 2.3 ContestTeam

职责：

- 保存一支参赛队伍的组织形态
- 作为赛事与成员之间的中间层

建议字段：

- `contest`
  所属赛事
- `team_name`
  队伍名称
- `members`
  队员列表，多对多关联 `MemberProfile`
- `coach_name`
  指导老师或带队老师，可选
- `leader`
  队长，可选
- `note`
  备注，可选

说明：

- 第一版中，一个成员可以参与多场比赛，也可以在不同赛事中属于不同队伍。
- 若存在个人赛场景，仍可以通过单人成队方式兼容，不必额外拆模型。

### 2.4 ContestResult

职责：

- 保存某支队伍在某场赛事中的最终成绩
- 作为 `Rating` 与 `Level` 的直接计算依据

建议字段：

- `contest`
  所属赛事
- `team`
  所属队伍
- `award_type`
  奖项类型
- `award_label`
  奖项展示文案
- `rank_label`
  名次描述，允许自由文本
- `rating_delta`
  本场为队员带来的基础积分变化，可由系统计算后落库
- `verified`
  是否已认证
- `verified_by`
  认证管理员
- `verified_at`
  认证时间
- `evidence_url`
  证据链接或附件路径，可选
- `note`
  备注，可选

建议枚举：

- `award_type`
  `gold / silver / bronze / honorable / finalist / participation / custom`

说明：

- `award_label` 用于前台最终展示，例如 `金奖`、`银奖`、`区域赛正式参赛`。
- `rank_label` 不要求严格数值化，可记录 `第 12 名`、`晋级省赛`、`校赛一等奖` 等管理端容易维护的描述。
- 第一版不包含 `solved_count` 字段。

### 2.5 MemberCompetitionProfile

职责：

- 保存队员当前竞赛能力摘要
- 为首页、个人页、榜单页提供高频读取数据

建议字段：

- `member`
  关联 `MemberProfile`
- `current_rating`
  当前积分
- `current_level`
  当前等级
- `peak_rating`
  历史最高积分
- `peak_level`
  历史最高等级
- `primary_color`
  展示颜色
- `highest_award_label`
  当前最高代表奖项
- `latest_contest_result`
  最近一条已认证赛事结果
- `last_calculated_at`
  最近重算时间

说明：

- 该表可视为缓存快照，真实依据仍来自 `ContestResult`。
- 第一版可采用“成绩变动后重算该成员档案”的方式，无需复杂异步体系。

## 3. 页面设计

### 3.1 Member 首页

目标：

- 在不破坏当前 `ACM Star` 设计的前提下，引入“竞赛实力”这一长期维度。

新增内容建议：

- 用户姓名按 `ACM Level` 显示颜色
- 在首页顶部信息区新增：
  - `Rating`
  - `Level`
  - 最高奖项
- 保留现有 `ACM Star` 卡片，继续代表近期活跃状态

推荐文案：

- `ACM Star`：近期参与状态
- `ACM Level`：竞赛等级
- `Rating`：赛事积分

展示示例：

- `Rating 1420 · Expert`
- `最高奖项：CCPC 省赛银奖`

### 3.2 Member 个人主页

新增一个“竞赛档案”模块，建议拆为三块：

1. 竞赛身份摘要
   展示 `Level`、`Rating`、历史最高等级、颜色名
2. 荣誉摘要
   展示最高奖项、最近一场赛事、最近一次晋级
3. 成绩时间线
   展示按时间倒序排列的赛事记录

成绩时间线每条记录建议展示：

- 赛事名称
- 时间
- 队伍名称
- 队伍成员
- 奖项文案
- 名次描述
- 认证状态

不展示：

- 解题数量

### 3.3 竞赛天梯页

目标：

- 形成可视化的成员梯队感知
- 强化等级体系的团队记忆点

页面建议包含：

- 当前等级分布统计
- 成员排行榜
- 最近晋级成员
- 按年级 / 专业 / 班级筛选

排行榜每行建议展示：

- 成员姓名
- 颜色名
- `Level`
- `Rating`
- 最高奖项
- 最近赛事

### 3.4 管理端赛事管理页

用于维护赛事基础信息。

建议能力：

- 创建赛事
- 编辑赛事
- 发布赛事
- 归档赛事
- 查看该赛事下所有队伍与成绩

列表字段建议：

- 赛事名称
- 系列
- 阶段
- 日期
- 级别
- 状态
- 已录入队伍数
- 已认证成绩数

### 3.5 管理端成绩录入页

用于快速录入和审核成绩。

建议支持：

- 选择赛事
- 创建或选择参赛队伍
- 批量选择成员
- 填写奖项与名次描述
- 提交认证
- 查看该记录将影响哪些成员的等级

录入表单建议字段：

- 赛事
- 队伍名称
- 队员列表
- 指导老师
- 奖项类型
- 奖项展示文案
- 名次描述
- 证据链接
- 备注

### 3.6 管理端天梯统计页

目标：

- 让管理员看见整体竞赛发展情况，而不仅是单个成员结果。

建议展示：

- 当前有等级成员数
- 各等级人数分布
- 近一学期新增获奖人数
- 近一学期新增晋级人数
- 高等级成员名单
- 可按届次、班级、专业筛选

## 4. 评分规则

### 4.1 设计原则

第一版评分规则需满足：

- 能解释
- 易录入
- 易调整
- 不依赖复杂外部数据

因此采用“赛事权重 + 奖项加分 + 参赛基础分 + 时间衰减”的混合方案，而非纯 Elo 模型。

### 4.2 Rating 构成

每条 `ContestResult` 对成员产生的积分变化建议由以下部分组成：

- `参赛基础分`
  参加正式赛事即可获得的基础积分
- `赛事级别权重`
  不同级别赛事对应不同倍率
- `奖项层级加分`
  奖项越高，加分越多
- `名次描述微调`
  可选，用于少量特殊情况
- `时间衰减`
  历史成绩保留，但影响逐步降低

### 4.3 建议基础口径

赛事级别建议权重：

- `national`：`1.6`
- `regional`：`1.4`
- `provincial`：`1.2`
- `campus`：`1.0`
- `internal`：`0.8`

奖项层级建议加分：

- `gold`：`+120`
- `silver`：`+80`
- `bronze`：`+50`
- `honorable`：`+25`
- `finalist`：`+15`
- `participation`：`+5`

参赛基础分建议：

- 正式参赛：`20`

示意公式：

`rating_delta = (base_participation_score + award_bonus) * level_weight * decay_factor`

说明：

- `base_participation_score` 默认为 `20`
- `award_bonus` 根据奖项类型决定
- `level_weight` 根据赛事级别决定
- `decay_factor` 根据比赛时间决定

### 4.4 时间衰减

为避免历史成绩永久固化，建议加入温和衰减：

- 最近 `12` 个月：`1.0`
- `12-24` 个月：`0.85`
- `24-36` 个月：`0.7`
- `36` 个月以上：`0.55`

说明：

- 衰减的是成绩影响，不删除历史记录。
- 前台仍可以完整展示历史获奖。

### 4.5 Level 划分

第一版建议采用离散等级制，同时保留颜色名展示。

建议等级：

- `Unrated`
  灰色，尚无已认证赛事成绩
- `Rookie`
  绿色
- `Solver`
  青色
- `Specialist`
  蓝色
- `Expert`
  紫色
- `Master`
  橙色
- `Legend`
  红色

建议积分区间：

- `Unrated`：`0`
- `Rookie`：`1 - 199`
- `Solver`：`200 - 499`
- `Specialist`：`500 - 899`
- `Expert`：`900 - 1399`
- `Master`：`1400 - 1999`
- `Legend`：`2000+`

说明：

- 该区间仅作为第一版初值，后续可根据真实成员分布调整。
- 用户前台同时展示 `Level` 文案和颜色，不必强制展示精确排名。

### 4.6 名次描述的使用方式

由于第一版不引入 `解题数量`，`rank_label` 仅作为展示补充与少量人工修正依据，不作为强依赖评分字段。

建议规则：

- 默认只根据赛事级别和奖项计算
- 若管理员认为某些成绩需要特殊微调，可额外设置 `manual_bonus`
- 前台继续展示 `rank_label`，提升履历可读性

### 4.7 计算策略

建议采用以下方式更新成员积分：

1. 一条 `ContestResult` 被创建或认证后
2. 找到该队伍下所有成员
3. 重算这些成员的全部已认证赛事结果总分
4. 更新 `MemberCompetitionProfile`

优点：

- 逻辑直观
- 容易调试
- 规则调整后支持整表重算

## 5. 管理流程

### 5.1 赛事录入流程

1. 管理员创建 `Contest`
2. 填写赛事基础信息，包括名称、系列、阶段、日期、级别
3. 系统根据级别给出默认权重
4. 管理员确认并发布赛事

### 5.2 队伍录入流程

1. 管理员进入某赛事详情页
2. 创建一个或多个 `ContestTeam`
3. 为每支队伍选择成员
4. 选填队长、指导老师和备注

### 5.3 成绩录入流程

1. 管理员选择某支队伍
2. 创建 `ContestResult`
3. 填写奖项类型、奖项文案、名次描述
4. 上传或填写证据链接，可选
5. 提交保存

### 5.4 认证与生效流程

1. 成绩初始可为未认证状态
2. 管理员核对证据材料后点击认证
3. 系统记录 `verified_by` 与 `verified_at`
4. 系统自动重算相关成员的 `Rating / Level`
5. 结果同步反映到成员主页和天梯页

### 5.5 修改与撤销流程

1. 若成绩信息录入错误，管理员可以编辑成绩
2. 若成绩应失效，管理员可以取消认证或标记作废
3. 系统重新计算受影响成员的积分与等级
4. 前台历史记录保留状态标记，避免无痕消失

### 5.6 推荐权限边界

- `Member`
  只可查看自己的竞赛档案与公开天梯信息
- `Admin`
  可创建赛事、录入队伍、录入并认证成绩
- `Super Admin`
  可维护等级阈值、赛事权重、奖项积分规则，并触发全量重算

### 5.7 第一版实施顺序

建议按以下顺序实现：

1. 新增 `Contest / ContestTeam / ContestResult`
2. 完成管理端赛事与成绩录入
3. 完成成员端竞赛档案展示
4. 完成 `Rating / Level` 计算与缓存
5. 完成竞赛天梯页与统计页

这样可以先保证“数据可录、结果可看”，再逐步增强榜单与运营能力。
